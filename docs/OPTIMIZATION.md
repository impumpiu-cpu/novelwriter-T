# Предложения по развитию и оптимизации NovWr

Документ состоит из трёх частей:

- **Часть I** — оценка запрошенных пользовательских функций (типизированные связи,
  группы тегов, горячие клавиши, режим технического писателя);
- **Часть II** — техническая оптимизация без изменения функционала
  (надёжность, скорость, память, сопровождаемость);
- **Часть III** — сводный план: все пункты рассортированы по группам к реализации.

---

## Часть I. Оценка запрошенных функций

### F1. Типизированные связи (сюжет, персонаж, предмет, локация…), как в novelWriter

**Текущее состояние.** Типизация уже есть на уровне *сущностей*: поле
`WorldEntity.entity_type` (Персонаж, Место, Фракция, Концепция, Транспорт, Предмет),
причём тип — свободная строка до 50 символов, и интерфейс уже позволяет вводить
**собственный тип** (поле «Свой тип...» в карточке сущности). *Связи* же
(`WorldRelationship`) типизированы слабее: есть свободная текстовая метка
(`label` + каноническая форма `label_canonical` для дедупликации), но нет
категории связи, по которой можно фильтровать дерево.

**Оценка: реализуемо, средняя сложность, хорошо ложится на существующую схему.**

Решение в две ступени:

1. **Категория связи** — добавить в `world_relationships` необязательную колонку
   `relation_kind` (например: `plot` — сюжетная, `social` — межперсонажная,
   `possession` — владение предметом, `spatial` — привязка к локации,
   `custom` — пользовательская). Миграция Alembic тривиальна (nullable-колонка),
   существующие данные не затрагиваются. В API (`app/schemas.py`) поле опционально —
   обратная совместимость сохраняется.
2. **Сюжет как узел дерева** — тип сущности «Сюжетная линия» (`Plot`) уже можно
   создать через «Свой тип» без единой строчки кода; для полноценной поддержки
   достаточно добавить его в предустановленный список (`labels.ts`,
   `ENTITY_TYPE_LABEL`) и в промпты извлечения bootstrap, чтобы ИИ находил
   сюжетные линии автоматически. Тогда связи «персонаж → участвует в → сюжетная
   линия», «предмет → ключ к → сюжетная линия» естественно входят в имеющееся
   дерево взаимосвязей.

Интеграция в интерфейс: фильтр/раскраска по `relation_kind` в списках связей
Atlas и в инспекторе связей; LoreManager при инъекции контекста может
приоритизировать сюжетные связи для текущей главы.

Объём работ: миграция + схема + CRUD (~1 день), UI-фильтры и подписи (~1–2 дня),
обновление промптов извлечения (~0,5 дня).

### F2. Пользовательские группы тегов (ключевые/второстепенные персонажи и т. п.)

**Текущее состояние.** Группировка сущностей отсутствует. Ближайшие механизмы:
`entity_type` (классификация по природе объекта, не по важности), `visibility`
(активна/справочная/скрыта — влияет на инъекцию, но это не группы) и
`LoreEntry.priority` (числовой приоритет инъекции, скрыт от пользователя).

**Оценка: реализуемо, средняя сложность, чисто аддитивная функция.**

Решение:

- Две таблицы: `world_entity_groups` (id, novel_id, name, color/sort_order) и
  связка many-to-many `world_entity_group_members` (group_id, entity_id).
  Many-to-many важно: один персонаж может состоять и в «Ключевые персонажи»,
  и в «Орден Небесного меча».
- Группы создаёт и наполняет пользователь вручную (как и запрошено) — CRUD-эндпоинты
  под `/api/novels/{id}/world/groups`, управление в навигаторе Atlas
  (создать группу, перетащить/отметить сущности, фильтр-чипы по группам).
- Побочная ценность: группе можно назначить вес инъекции (члены «Ключевых
  персонажей» получают приоритет в контексте продолжения) и использовать группы
  как область поиска Copilot («ищи только среди второстепенных персонажей»).

Объём работ: модель + миграция + API (~1–2 дня), UI навигатора и фильтров
(~2–3 дня). Функция не затрагивает существующие данные и промпты.

### F3. Горячая клавиша присвоения тега с автоподсказкой (по образцу Ctrl+T)

**Текущее состояние.** Системы горячих клавиш в редакторе нет (только Enter/
Shift+Enter в панели Copilot). Механика «тега для индексации» в терминах NovWr —
это **псевдоним сущности** (`aliases`) и **ключевое слово лорбука** (`LoreKey`):
именно по ним автомат Ахо–Корасика находит упоминания и внедряет сеттинг
в контекст. То есть бэкенд-часть функции уже существует полностью.

**Оценка: реализуемо, низкая-средняя сложность, преимущественно фронтенд.**

Решение:

- В редакторе главы (`ChapterEditor`) по горячей клавише открывать **палитру тегов**
  (combobox): выделенный текст подставляется как кандидат; список автоподсказок —
  существующие сущности и их псевдонимы (данные уже загружены в Atlas-хуках);
  выбор существующей сущности добавляет выделение как её псевдоним, а пункт
  «Создать…» заводит новую черновую сущность с этим именем. Используются
  существующие эндпоинты обновления/создания сущностей.
- ⚠️ Именно **Ctrl+T перехватывается браузером** (новая вкладка) и не может быть
  переопределено веб-приложением. Рекомендуемые варианты: `Ctrl+K`
  (де-факто стандарт командных палитр), `Alt+T` или `Ctrl+Shift+E`;
  привязку стоит сделать настраиваемой (хранить в `localStorage` рядом с темой
  и локалью) и показать подсказку в настройках.
- Инфраструктурно: небольшой хук `useHotkey(combo, handler)` с проверкой фокуса,
  чтобы клавиша работала и в режиме чтения, и в режиме правки.

Объём работ: палитра + хук + интеграция с API (~2–3 дня).

### F4. NovWr как ассистент технического писателя (параметры насосов, ёмкости датчиков, свойства веществ)

**Оценка: ядро подходит уже сейчас; для продуктивного использования нужны
доработки среднего масштаба.**

Что уже работает «из коробки» для технических текстов:

- **Модель мира = база знаний оборудования.** «Насос Н-201» — сущность
  пользовательского типа «Оборудование», его параметры — атрибуты
  (`подача: 50 м³/ч`, `напор: 80 м`); «серная кислота» — сущность «Вещество»
  с атрибутами свойств. Пара «внешнее/истина» у атрибутов пригодна для пары
  «паспортное значение / фактическое по обследованию».
- **Связи** описывают технологическую схему: «насос → подаёт в → ёмкость Е-3»,
  «датчик → контролирует → температуру реактора».
- **Системы** (иерархия/хронология/список) подходят для классификаторов
  оборудования, регламентов пуска (хронология) и перечней требований (список).
- **Copilot и индекс всей книги** дают контроль текста: «где в документе
  упоминается давление насоса Н-201?» — ответ с цитатами-источниками.
- **Предупреждения о дрейфе** уже ловят термины, отсутствующие в модели мира.

Необходимые доработки (по возрастанию объёма):

1. **Предустановленный «технический» набор типов** — worldpack-шаблон с типами
   Оборудование/Вещество/Датчик/Документ/Требование и типовыми атрибутами.
   Механизм импорта worldpack уже есть. (~1–2 дня)
2. **Типизированные значения атрибутов** — сейчас атрибуты строковые; нужны
   число + единица измерения (+ допуск), чтобы сравнивать значения, а не строки.
   Колонки `value_numeric`, `unit` в `world_entity_attributes` + валидация. (~3–5 дней)
3. **Проверка числовой согласованности** — расширить постпроверку: если параметр
   сущности упомянут в тексте со значением, отличным от модели мира
   («подача 60 м³/ч» при канонических 50), выдавать предупреждение дрейфа.
   Основа (сопоставление упоминаний через LoreManager) уже есть. (~1 неделя)
4. **Импорт табличных данных** — загрузка CSV/XLSX с параметрами прямо
   в атрибуты сущностей (обход текстового bootstrap). (~3–5 дней)
5. **Профиль промптов «технический документ»** — текущие промпты продолжения
   и Copilot заточены под художественную прозу; нужен второй профиль
   (нейтральный стиль, запрет «творческих» дополнений фактов) и переключатель
   жанра на уровне проекта. Реестр промптов уже поддерживает варианты. (~1 неделя)
6. **Парсинг структуры разделов** — заголовки вида «3.2.1 Насосное оборудование»
   вместо «Глава N»; парсер расширяется дополнительным профилем заголовков. (~2–3 дня)

Вывод: как «база знаний + поиск с цитатами + контроль терминологии» репозиторий
пригоден для технического писателя уже сейчас (достаточно пункта 1);
полноценный контроль числовых параметров даёт связка пунктов 2–3.

## Часть II. Техническая оптимизация (без изменения функционала)

### Надёжность

- **SQLite под конкурентной записью**: включить `PRAGMA journal_mode=WAL` и
  `busy_timeout` (~5000 мс) при инициализации соединения — worker и API делят
  одну базу, это снижает риск `database is locked`. Для PostgreSQL — явные
  `pool_size`, `max_overflow`, `pool_pre_ping=True`.
- **Устаревший адаптер datetime SQLite** (DeprecationWarning на Python ≥ 3.12):
  зарегистрировать явные адаптеры/конвертеры, пока предупреждение не стало ошибкой.
- **Ретраи LLM-вызовов**: экспоненциальный повтор (2–3 попытки с джиттером)
  для идемпотентных запросов (bootstrap, Copilot-анализ) в `ai_client`.
- **Грейсфул-шатдаун worker'а**: обработчик SIGTERM, который дожидается конца
  текущей задачи и освобождает аренду — быстрее перезапуски контейнера.
- **Health-check очереди задач**: метрика «возраст самой старой невзятой задачи»
  в `/api/health/access` — раннее обнаружение остановившегося worker'а.

### Скорость

- **Кеширование настроек**: убедиться, что `get_settings()` в горячих путях
  всегда кеширован, перечитывание — только через `reload_settings()`.
- **LoreManager**: инкрементальное обновление автомата Ахо–Корасика или отложенная
  пересборка с дебаунсом (сейчас каждое подтверждение черновика инвалидирует кеш
  целиком; болезненно на 10⁴–10⁵ записей).
- **Пагинация списков API**: сущности/связи возвращаются целиком; серверная
  пагинация или проекция полей ускорит первый рендер Atlas на больших мирах.
- **Дорожки параллелизма LLM**: динамическое разделение лимитов интерактивных
  и фоновых вызовов (сейчас константы `MAX_CONCURRENT_LLM_CALLS` /
  `MAX_BACKGROUND_CONCURRENT_LLM_CALLS`).
- **Размер бандла фронтенда**: `index-*.js` ~342 КБ (108 КБ gzip); ленивая
  загрузка `@xyflow/react` (нужен только устаревшим графовым системам),
  `framer-motion` на главной и **пакетов локалей** (динамический `import()`
  по выбранной локали — сейчас все три локали грузятся всем).
- **Инкрементальная пересборка индекса книги**: пересобирать только окна
  затронутых глав, а не весь индекс (важно на романах в миллионы знаков).

### Память и ресурсы

- **Стриминг импорта файлов**: потоковое декодирование `.txt` вместо чтения
  целиком (30 МБ × параллельные загрузки).
- **LRU-лимит кеша LoreManager**: кеш автоматов на процесс не ограничен по числу
  романов — в hosted-режиме нужен предел.

### Сопровождаемость

- **Единый конверт ошибок API**: часть эндпоинтов возвращает `detail` строкой,
  часть — объектом с `code`; фронтенд (`llmErrorMessages.ts`) вынужден разбирать оба.
- **Тест полноты локалей**: снапшот-тест «все ключи есть во всех локалях»
  (сейчас полноту ru/zh гарантирует только типизация, en-каталог частичный).
- **Устаревшие регэкспы трейсов Copilot** в `NovelCopilotResearchProcess.tsx`
  дублируют серверный каталог сообщений — удалить после стабилизации ключей.

## Что уже сделано в рамках рефакторинга русификации

- `raise ... from exc / from None` во всех обработчиках исключений API — точные
  цепочки причин в логах.
- `zip(..., strict=True)` в компиляции предложений Copilot — инвариант длины
  списков проверяется, а не молча обрезается.
- Переключатель языка в настройках построен из `SUPPORTED_UI_LOCALES` —
  добавление новой локали больше не требует правки компонента.
