# Предложения по оптимизации NovWr

Документ фиксирует возможные направления оптимизации **без изменения функционала**.
Разделы упорядочены по соотношению «польза / риск».

---

## 1. Надёжность

### 1.1. Пул соединений и повторные попытки при работе с БД
- Сейчас worker и веб-процесс делят одну SQLite-базу; при конкурентной записи возможны
  ошибки `database is locked`. Стоит включить `PRAGMA busy_timeout` (например, 5000 мс)
  и режим WAL (`PRAGMA journal_mode=WAL`) при инициализации соединения SQLite —
  это резко снижает вероятность конфликтов записи между worker'ом и API.
- Для PostgreSQL — задать явные параметры пула (`pool_size`, `max_overflow`,
  `pool_pre_ping=True`), чтобы «мёртвые» соединения не приводили к 500-м после простоя.

### 1.2. Устаревший адаптер datetime в SQLite
- Тесты выдают `DeprecationWarning: The default datetime adapter is deprecated`
  (Python ≥ 3.12). Стоит зарегистрировать явные адаптеры/конвертеры datetime
  (или перейти на хранение ISO-строк/UTC-timestamp), пока предупреждение
  не превратилось в ошибку в будущих версиях Python.

### 1.3. Ретраи запросов к LLM
- В `ai_client` сетевые сбои приводят к немедленному `LLMUnavailableError`.
  Экспоненциальный повтор (2–3 попытки с джиттером) для идемпотентных вызовов
  (структурированный вывод bootstrap, Copilot-анализ) повысил бы устойчивость
  к кратковременным сбоям шлюзов без изменения поведения при постоянных ошибках.

### 1.4. Грейсфул-шатдаун worker'а
- `run_worker_loop` завершает процесс без обработки SIGTERM: задача, прерванная
  посреди исполнения, будет ждать истечения аренды (lease). Обработчик сигнала,
  который дожидается конца текущей задачи и освобождает аренду, ускорит
  перезапуски при обновлениях контейнера.

### 1.5. Health-check фоновых задач
- `/api/health/access` проверяет БД и статику, но не состояние очереди задач.
  Полезно добавить метрику «возраст самой старой невзятой задачи» — раннее
  обнаружение остановившегося worker'а (сейчас это видно только по зависшему импорту).

## 2. Скорость

### 2.1. Кеширование настроек и каталогов
- `get_settings()` вызывается в горячих путях по многу раз за запрос. Проверить, что
  везде используется кешированная версия (lru_cache), а перечитывание — только через
  `reload_settings()`.

### 2.2. Инъекция сеттинга (LoreManager)
- Автомат Ахо–Корасик уже кешируется, но пересобирается целиком при любом изменении
  записи мира. Для больших миров (10⁴–10⁵ записей) выгоднее инкрементальное
  обновление либо отложенная пересборка с дебаунсом (сейчас каждое подтверждение
  черновика в Atlas инвалидирует кеш).

### 2.3. Пагинация тяжёлых списков API
- Эндпоинты списков сущностей/связей возвращают всё разом; Atlas на большом мире
  получает сотни килобайт JSON. Серверная пагинация или `fields=`-проекция
  сократили бы время первого рендера Atlas.

### 2.4. Параллелизм генерации версий
- Версии продолжения уже генерируются конкурентно, но при `MAX_CONCURRENT_LLM_CALLS=4`
  два пользователя по 2 версии исчерпывают лимит. Стоит разделить лимиты на
  «интерактивные» и «фоновые» дорожки динамически (сейчас константы), чтобы фоновый
  bootstrap не задерживал интерактивные продолжения в selfhost.

### 2.5. Фронтенд: размер бандла
- `index-*.js` ~342 КБ (108 КБ gzip). Основные кандидаты на lazy-загрузку:
  `@xyflow/react` (граф — только для устаревших графовых систем) и `framer-motion`
  на маркетинговой главной. Также стоит подключать пакеты сообщений локалей
  динамически (`import()` по выбранной локали) — сейчас все три локали
  загружаются всеми пользователями.

### 2.6. Индексация всей книги
- Rust-модуль уже быстрый; узкое место — полная пересборка индекса при изменении
  одной главы. Инкрементальная пересборка только затронутых окон (по номеру главы)
  снизит фоновые задержки на романах в миллионы знаков.

## 3. Память и ресурсы

### 3.1. Стриминг импорта файлов
- Импорт `.txt` читает файл целиком в память; для файлов по 30 МБ в нескольких
  параллельных загрузках это заметно. Потоковое декодирование и парсинг по
  главам ограничили бы пиковое потребление.

### 3.2. Ограничение кеша LoreManager
- Кеш автоматов на процесс не ограничен по числу романов; в hosted-режиме с сотнями
  активных романов уместен LRU-лимит.

## 4. Сопровождаемость (низкий риск)

### 4.1. Единая точка форматирования сообщений об ошибках API
- Часть эндпоинтов возвращает `detail` строкой, часть — объектом с `code`.
  Единый конверт ошибок упростил бы фронтенд-обработку (`llmErrorMessages.ts`
  уже вынужден разбирать оба варианта).

### 4.2. Тесты локалей
- Добавить снапшот-тест «все ключи присутствуют во всех локалях» (сейчас полноту
  ru/zh гарантирует только типизация TypeScript, а en-каталог частичный —
  недостающие ключи молча падают в zh).

### 4.3. Устаревшая нормализация трейсов Copilot на фронтенде
- `NovelCopilotResearchProcess.tsx` перезаписывает китайские строки трейса
  регулярными выражениями — это дублирует серверный каталог сообщений.
  После стабилизации серверных ключей эти регэкспы можно удалить.

## Что уже сделано в рамках этого рефакторинга

- `raise ... from exc / from None` во всех обработчиках исключений API — точные
  цепочки причин в логах.
- `zip(..., strict=True)` в компиляции предложений Copilot — инвариант длины
  списков проверяется, а не молча обрезается.
- Переключатель языка в настройках построен из `SUPPORTED_UI_LOCALES` —
  добавление новой локали больше не требует правки компонента.
